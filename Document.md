# 문서 레이아웃 규격 — 참고 PDF 분석 및 작성 가이드

본 문서는 다음 두 PDF를 텍스트 추출 기준으로 분석한 내용과, 이후 동일 규격으로 문서를 작성하는 방법을 정리한다.

- `_712020f2987b5aa1f54d9aa3fb6ede504f9b3a-복잡도 단계별 적 레퍼런스 가이드-180426-073327.pdf` (적 레퍼런스 가이드)
- `_712020f2987b5aa1f54d9aa3fb6ede504f9b3a-프로토타이핑 명세서-180426-073509.pdf` (프로토타이핑 명세서)

시각적 그리드(여백·컬럼·색상)는 추출본만으로는 확인하지 못했으며, **정보 구조와 반복 패턴**을 규격의 기준으로 삼는다. **웹 버전 Enemy Reference Guide**(접기/펼치기·외부 링크 버튼)는 **§4**에서 스크린샷 기준으로 정리한다.

## 1. 문서 A — 복잡도 단계별 적 레퍼런스 가이드

**역할:** 난이도·복잡도 **축**을 세우고, 단계마다 **추천 레퍼런스(적)**를 카드 형태로 나열한다.

### 구조 패턴

- **표지 블록:** 한글 제목 → 영문 부제 → 한 줄 설명(범위·기준 게임).
- **큰 축(단계 번호):** 단계 번호 + **단계명** + **프로토타입 유형** + **한 줄 정의**(콤마로 나열한 키워드).
- **선행 조건:** 상위 단계 요약(예: “1단계 전부” + 나열된 검증 항목).
- **엔트리(적) 단위:**  
  한글명 + 영문명 → **우선순위**(1순위 등) → **외부 링크 줄** → **추천 이유** → (선택) **예상 패턴** → **테스트 커버리지**(짧은 라벨 나열) → **주의/한계** 한 문단.
- **푸터:** `문서명 • 기준 • 연도` + 페이지 `n of N`.

### 레이아웃 규격화 포인트

- “단계”는 **항상 같은 순서**(번호 → 제목 → 한 줄 정의 → 선행 조건).
- 각 적은 **동일한 필드 세트**(이름 쌍, 우선순위, 링크, 이유, 패턴, 커버리지, 캐브닛)로 **카드 템플릿**화한다.

## 2. 문서 B — 프로토타이핑 명세서

**역할:** 시스템별 **요구사항 + 파라미터 성격**을 **번호 매긴 목차**로 정의한다.

### 구조 패턴

- **표지 블록:** 제목 → 부제(“구현 명세서”) → **한 줄 스코프**.
- **요약 대시보드:** 핵심 수치(시스템 수, 명세 항목 수), **기준 레퍼런스**, **파라미터 타입 범례**(frame / value / bool / enum).
- **시스템 단위:** `번호 + 시스템명 + 항목 수 + ▶` 등 **접힘/펼침 시각 단서**.
- **하위 명세:** 짧은 제목(한 줄 개념) + 본문(수치·프레임·정책).
- **강조 블록:** “핵심 기술 결정 사항”, “가장 버그가 많은 영역”처럼 **리스크/결정**만 따로 박스.
- **범위 제외/의도적 생략:** 마지막에 **참고사항**으로 몰아넣기.
- **푸터:** 도구/엔진 · 연도.

### 레이아웃 규격화 포인트

- 상단 **한 줄 메타 + 수치 + 범례**는 기술 명세서 공통 블록으로 둔다.
- 본문은 **시스템 ID 고정 자릿수(01, 02…)**로 정렬감을 유지한다.

## 3. 이후 문서 작성 시 규격화 실무 방법

### (1) 문서 유형별 마스터 템플릿 2종

| 유형 | A형(가이드/레퍼런스) | B형(명세/요구사항) |
|------|----------------------|---------------------|
| 상단 | 제목·부제·한 줄 범위 | + **수치 요약·범례** |
| 본문 | **단계 → 카드 반복** | **번호 시스템 → 항목 리스트** |
| 끝 | 푸터 고정 문구 | + **의도적 생략/참고** |

워드/구글독스/노션 중 하나를 골라 **“새 문서 = 템플릿 복제”**로 표지·푸터·제목 스타일이 이미 들어간 파일을 유지한다.

### (2) 스타일 시트(이름 통일)

- `Title` / `Subtitle` / `H1 단계` / `H2 카드제목` / `Body` / `Caption·푸터` / `Callout(결정·주의)`  
도구에서 **스타일 이름을 고정**하고, 본문은 수동 서식만 쓰지 않는다.

### (3) 카드형(A형) 필드 스키마

스프레드시트나 노션 DB 한 줄 = 한 카드.

예시 열: `단계번호 | 단계명 | 엔티티명KO | 엔티티명EN | 우선순위 | 링크 | 이유 | 패턴 | 커버리지태그 | 주의`

### (4) 명세형(B형) 항목 스키마

예시 열: `시스템ID | 시스템명 | 항목ID | 항목명 | 설명 | 파라미터타입 | 기본값/범위 | 비고`

프레임/수치/bool/enum은 **한 열**에만 기록해 구현·튜닝 테이블로 이전하기 쉽게 한다.

### (5) 페이지 푸터 규칙

`{문서 짧은 이름} • {기준/엔진/프로젝트} • {연도}` + `페이지 n / N`

### (6) 도구별 추천

- **협업·빠른 반복:** 구글 독스(템플릿 + 스타일) 또는 노션(데이터베이스 + 갤러리/보드 뷰).
- **인쇄·제출 PDF:** 워드 서식 템플릿 → PDF; 또는 **Markdown + Pandoc** + `reference.docx`.
- **시각 그리드까지 동일:** 인디자인/피그마에 **한 페이지 마스터** 복제.

### (7) 초안 완료 시 체크리스트

- 표지에 **한 줄 스코프**가 있는가.
- A형이면 **모든 카드에 같은 소제목 세트**가 있는가.
- B형이면 **상단 범례 + 시스템 번호 자릿수**가 있는가.
- **결정/리스크/생략 범위**가 본문과 섞이지 않고 블록으로 구분되는가.
- 푸터·페이지 번호가 **전 페이지 동일 규칙**인가.

## 4. 웹 UI — 스크린샷 기준 (접기/펼치기·링크 버튼)

동일 **A형(적 레퍼런스 가이드)** 콘텐츠를 웹으로 옮긴 화면 기준이다. PDF와 **정보 단위는 같고**, 표현만 **아코디언 + 배지 + 칩 + 외부 링크 버튼**으로 나뉜다. 아래는 **기본 패턴형(단계 1)** 화면과 **정직한 보스형·이후 단계(단계 2·3 등)** 화면을 함께 반영한 규격이다.

### 4.1 페이지 머리글

| 요소 | 규격 |
|------|------|
| 메인 타이틀 | 영문 대문자 중앙 강조 (예: ENEMY REFERENCE GUIDE) |
| 부제 | 한 줄 스코프(복잡도 단계별 · 기준 게임 등) |

### 4.2 단계 블록 (레벨 1 — 접지 않는 구역)

한 **복잡도 단계**마다 아래 순서를 고정한다.

| 순서 | UI 요소 | 내용 |
|------|---------|------|
| 1 | 단계 번호 | 원형 배지 안 숫자. **단계별로 배지 색이 달라질 수 있음** (예: 1단계 초록, 2단계 파랑, 3단계 주황) — 복잡도 구간 시각 구분 |
| 2 | 단계명 + **개발 단계 배지** | 제목 + 옆 작은 라벨. 라벨은 **콘텐츠에 따라 변경** (예: `프로토타입`, `본개발 1차`) |
| 3 | 한 줄 설명 | 카테고리 특성 요약 문장 (보조 회색 톤) |
| 4 | 특성 칩(태그) | 가로 나열 pill — 기술·검증 초점 (예: 액션 페이즈·히트박스·기본 AI·거리 분기 / 콤보 체인 AI·강인도 등) |

→ PDF의 「단계 번호 + 단계명 + 정의 + 키워드 나열」과 **1:1 대응**. 웹에서는 **원형 번호 색 + 개발 단계 배지**가 추가 시각 채널이므로 데이터 토큰(`stageBadgeTheme`, `stagePhaseLabel` 등)으로 명시한다.

#### 4.2.1 색·타이포 토큰 (보스형·다단계 스크린샷 기준)

디자인 시트·CSS 변수를 맞출 때 참고. 실제 색값은 프로젝트 팔레트에 맞게 조정한다.

| 용도 | 규칙 |
|------|------|
| 초기·프로토 구간 | 녹·청 계열 — 단계 번호 원, `프로토타입` 배지, **1순위 추천** 등 강조 |
| 본개발 이후 구간 | 주황 등 대비색 — 단계 번호 원, `본개발 1차` 류 배지로 일정·난이도 단계 구분 |
| 추천·우선순위 | 파란 톤 배지 (예: 1순위 추천) |
| 예상 패턴 칩 | **회색 계열** pill — 공격·행동 나열 |
| 테스트 커버리지 칩 | **녹색 계열** pill — 검증 항목을 패턴 칩과 색으로 분리 |
| 본문·부제 | 제목 대비 작고 명확한 위계 |
| 카드·요약 박스 | 연한 회색 컨테이너; 하단 박스는 **비용·리그·에셋** 등 기술 메모 |

### 4.3 적 카드 (레벨 2 — 아코디언)

**한 적 = 한 카드.** 접힘/펼침 **두 상태**를 명시해 두면 구현·디자인 시트가 맞는다.

#### 접힘 (collapsed)

| 영역 | 규격 |
|------|------|
| 본문 | **이름만** (한글 + 영문 한 줄) |
| 우측 | **펼치기** 아이콘 (원 안 오른쪽 화살표 등) |

#### 펼침 (expanded)

| 순서 | UI 요소 | 내용 |
|------|---------|------|
| 1 | 헤더 행 | 좌: 이름 쌍 / 우: **우선순위 배지** (예: 1순위 추천) + **접기** 아이콘 (아래쪽 삼각 등) |
| 2 | 링크 버튼 행 | 외부 자료 **개별 버튼** (테두리·텍스트 강조). 예: Fextralife Wiki, 나무위키, YouTube — **각각 `href` + 라벨** 스키마로 관리 |
| 3 | 소제목 + 본문 | **추천 이유** (단락 텍스트) |
| 4 | 칩 그룹 A | **예상 패턴** (회색 계열 pill, 다중) |
| 5 | 칩 그룹 B | **테스트 커버리지** (다른 색 pill로 시각 구분) |
| 6 | 요약 박스 | 좌측 세로 액센트 + 연한 배경 — **한 줄 결론/제작 관점 요약**, 비용·에셋·리그 공유 등 **기술 메모**(PDF의 주의·장점 문단과 대응) |

**한 단계 안에 적 여러 명:** 첫 카드만 펼침·나머지 접힘처럼 **동일 단계 블록 아래**에 카드를 세로로 쌓는다 (예: 펼침 1개 + 접힘 N개). 각 카드는 동일 §4.3 규격을 따른다.

### 4.4 인터랙션·접근성 규격 (구현/QA 체크용)

- **아코디언:** 헤더 전체 또는 전용 버튼이 `aria-expanded`, 키보드 포커스 순서, 아이콘만으로 상태가 구분되는지.
- **링크 버튼:** 실제로는 `<a>` 권장(새 탭 정책 `rel` 일관), 라벨·URL을 **데이터(JSON/CSV)**에서 분리해 PDF·웹 공통 소스로 쓸 수 있게 함.
- **칩:** 스크린 리더에서는 리스트 또는 쉼표 구분 문장으로 읽히게 할지 정책 결정.

### 4.5 PDF / 웹 공통 데이터 스키마 (권장)

웹 카드 1장 = DB/API 1건으로 두면 PDF 생성·정적 페이지·React가 같은 소스를 쓴다.

| 필드 | 용도 |
|------|------|
| `stageId` | 단계 번호 |
| `stageTitleKo` / `stageTitleEn` | 단계명(선택) |
| `stageSummary` | 한 줄 설명 |
| `stageTags[]` | 단계 특성 pill |
| `stageBadgeTheme` | 원형 번호 배지용 테마 토큰 (예: `green` / `blue` / `orange`) — UI만, 비즈니스 로직과 분리 |
| `stagePhaseLabel` | 단계 옆 배지 문구 (예: `프로토타입`, `본개발 1차`) |
| `stagePhaseStyle` | 배지 색·스타일 변형용 키 (선택, `stageBadgeTheme`과 연동 가능) |
| `enemyId` | 적 고유 ID |
| `nameKo` / `nameEn` | 카드 제목 |
| `priority` | 1순위 등 (없으면 null) |
| `links[]` | `{ label, url }` — 링크 버튼 행 |
| `reasonMd` 또는 `reason` | 추천 이유 |
| `patterns[]` | 예상 패턴 칩 |
| `coverage[]` | 테스트 커버리지 칩 |
| `summaryBox` | 하단 강조 박스 문단 |
| `defaultExpanded` | 첫 방문 시 펼칠 카드만 true 등 |

## 5. 한 줄 요약

두 PDF는 **「표지 메타 + 단계/시스템으로 뼈대를 나누고, 안에서는 카드 vs 항목 리스트로 반복」**이라는 공통 골격을 가진다. **A형(가이드)·B형(명세) 템플릿**과 **스타일 이름·DB 열 이름**을 고정하면 내용만 바꿔도 동일 규격으로 재사용하기 좋다. **웹 A형**은 그와 동일한 콘텐츠를 §4처럼 **아코디언 + 링크 행 + 이중 칩 + 요약 박스**로 매핑하면 된다. 단계가 올라갈수록 **번호 배지 색·개발 단계 배지 문구**가 바뀌어도 §4.2·§4.2.1·스키마 토큰으로 규격을 유지한다.
